Java Basis

Variables

In simple words ,Variables are the name of the memory location where we can store data.

In the Java Programming Language defines the following type of variables:

1). Instance Variables (Non-Static Fields)

This are fields declared without the keyword "Static" in it. Non-static fields are also known as "Instance variable" because there value are unique to each instance of the class.

2). Class Variables (Static Fields)

This are the fields declared with the "Static" modifier in it. This tell the compiler that there are exactly one copy of this variable and it is been shared among all the instance of the class.A field defining the number of gears for a particular kind of bicycle could be marked as static since conceptually the same number of gears will apply to all instances. The code static int numGears = 6; would create such a static field. Additionally, the keyword final could be added to indicate that the number of gears will never change.

3). Local Variables

The variable which are declared as the argument to the method or within the opening and the closing bracket of the method is known as local variable. There is no special keyword assigned for the local Variable, the determination comes entirely on the position of the variable with declared. As such, local variables are only visible to the methods in which they are declared; they are not accessible from the rest of the class.

Naming Convention of Variable

Variable name should not start with "$". Additionally, the dollar sign character, by convention, is never used at all. A similar convention exists for the underscore character; while it's technically legal to begin your variable's name with "_", this practice is discouraged. White space is not permitted.

The name you choose must not be a keyword or reserved word.

If the name you choose consists of only one word, spell that word in all lowercase letters. If it consists of more than one word, capitalize the first letter of each subsequent word. The name rahul and rahulSahare the prime example of this convention.

Basic Architecture Diagram of JVM

The diagram shows the basic architecture of the JVM. There are three main components of the Java Virtual Machine. They are as follows:

1). Class Loader SubSystem

2). JVM memory area

3). Execution engine

As we had already discuss that JVM is used for two basis operation i.e, load and run. So for loading the Java class some component is required, this component is class loader sub system. The class loader subSystem will load the Java classes in the JVM memory area. So when the ".class" file gets loaded in the memory area.Then who is responsible for running this loaded class?. It is "execution engine".

Class Loader SubSystem is responsible to load classes into JVM memory.

Class loader SubSystem will load the ".class" file in method area.

There are total of 5 memory subsection present in the memory area.The brief information is been given below:

1). Method Area - ".class" file is been loaded here.

2). Heap Area - The objects will be saved here.

3). Stack Area - The local variable and the method calls is been stored here.

4). PC register Area - Current executing instruction address is stored here.

Execution engine is responsible to run ".class" file line by line.

Something when the execution engine executes some native libraries are required. The native libraries consist of the native method, it is the method which are not been implemented in the Java. It may be implemented in some other languages like C++,C etc. Ex: "HashCode" or "HashClone" method is not been implemented in Java so this method implementation is taken from "native method libraries". Since this method is been written is some other language and it follows different set of instruction the interface is required in between the "execution engine" and the "native libraries method". This interface is "Java Native Interface".

Class Loader SubSystem

Class Loader SubSystem is the 1st component of the JVM to react while loading and running the Java based application.

It is responsible for 3 activites:

1).Loading

2).Linking

  • Verification
  • Preparation
  • Resolution

3).Initialization


Loading

Loading Means reading the Class Files and Store Corresponding Binary Data in the Method Area.


For Each Class File JVM will store the following Information in the Method Area

1).Fully Qualified Name of the Loaded Class OR Interface OR ENUM.

2).Fully Qualified Name of its Immediate Parent Class

3).Whether .class File is related to a Class Or Interface or ENUM.

4).The Modifier Information.

5).Variable or Field Information.

6).Methods Information.

7).Constant Pool Information and so on.


After loading .class File JVM will Immediately Create an Object of the Type Class to represent the Class Level Binary Information in the Method Area

Verifying

Check whether or not the read class is configured as described in the Java Language Specification and JVM specifications. This is the most complicated test process of the class load processes, and takes the longest time. Most cases of the JVM TCK test cases are to test whether or not a verification error occurs by loading wrong classes.

Preparing

Prepare a data structure that assigns the memory required by classes and indicates the fields, methods, and interfaces defined in the class.

Resolving

Change all symbolic references in the constant pool of the class to direct references.

Initializing

Initialize the class variables to proper values. Execute the static initializers and initialize the static fields to the configured values.

The JVM specification defines the tasks. However, it allows flexible application of the execution time.

Type of Class Loaders:

Every Class Loader SubSystem will have three class loader components. They are:

1). BootStrap Class Loader/Primordial Class Loader

2). Extension Class Loader

3). Application Class Loader/System CLass Loader

How Java Class Loader Works?

When a class loader is requested for class load, it checks whether or not the class exists in the class loader cache, the parent class loader, and itself, in the order listed. In short, it checks whether or not the class has been loaded in the class loader cache. If not, it checks the parent class loader. If the class is not found in the bootstrap class loader, the requested class loader searches for the class in the file system.



Bootstrap class loader

This CLass Loader is responsible for loading all the "Core Java API" classes.i.e; the classes present in the rt.jar .

The location of the rt.jar is jdk/jre/lib/rt.jar

BootStrap ClassLoader is by default available in JVM. It is been implemented in native languages like "C" and "C++".

Extension class loader

It is the child of the "BootStrap ClassLoader". It is responsible to load all the class present in the extension class path. It loads various security extension functions. Location of the extension class path is jdk/jre/lib/ext

Extension class loader is implemented in Java and the corresponding ".class" is sun.misc.launcher$extClassLoader.class

Application or System class loader

It is the child of the Extension Class Loader.Application Class Loader is responsible to load all the classes present in the Environment Variable Class Path.It loads the application classes. It loads the class in the $CLASSPATH specified by the user.

Extension class loader is implemented in Java and the corresponding ".class" is sun.misc.launcher$AppClassLoader.class

User-defined class loader

This is a class loader that an application user directly creates on the code.

Various Memory Areas of JVM

The memory area allocated by JVM is also known as "Runtime Data Area". It is the memory assigned when the JVm program runs on the OS. The runtime data area can be divided into 6 areas. Of the Six, one PC register, JVM Stack and Native Method Stack for each thread . Heap, Method Area and the runtime Constant Pool are shared by all threads.

1). PC Register - It is the register existing for each thread. It stores the address of the current instruction unless it is native. If the current method is native then the PC is undefined. All the CPU has PC, typically PC is incremented after each instruction and therefore it hold the address of the next instruction to be executed. The JVM uses the PC to keep track of where it is executing instructions, the PC will be infact pointing to the memory address in the method area.



2). JVM Stack Area - One JVM stack exist for one thread, and is created when the thread starts. The stack is based on the concept of "LIFO(Last In First Out)". The stack is made up of the stack frame. The JVM uses or pop the stack frame onto the JVM Stack. If any exception occurs while executing in the stack it is stored in the "printStackTrace()" method.

Whenever the method is been executed in the JVM, one Java Frame is been created for the thread. When the method call finishes the Java Stack is removed or deallocated. Each stack frame has the reference for the Local Variable, Operand Stack and runTime constant pool of a class where the method being executed belongs. The size of the local variable and the operand stack is fixed according to the method.

3). Native Method Stack - A stack of the native written language other than Java. In other word, it is the Stack used to execute C/C++ code invokde through JNI(Java Native Interface). According to the language, a "C" native stack or "C++" native stack can be created.

4). Method Area - The method area is shared by all threads, created when the JVM starts. The class loader initially loads the Class Files as corresponding Binary Data in the Method Area. The following information is been stored in the Method Area

i).Fully Qualified Name of the Loaded Class OR Interface OR ENUM.

ii).Fully Qualified Name of the Parent Class

iii).Whether .class File is related to a Class Or Interface or ENUM.

iv).The Modifier Information.

v).Variable or Field Information.

vi).Methods Information

vii).Constant Pool Information and so on.

5).Runtime constant pool: An area that corresponds to the constant_pool table in the class file format. This area is included in the method area; however, it plays the most core role in JVM operation. Therefore, the JVM specification separately describes its importance. As well as the constant of each class and interface, it contains all references for methods and fields. In short, when a method or field is referred to, the JVM searches the actual address of the method or field on the memory by using the runtime constant pool.

It is the space where the instances or the object is been stored and is the target of the garbage collection. It is the space provided for the dynamic allocation of the object at the runtime. This space is most frequently discussed when discussing issues such as JVM Performance. JVM vendors can determine how to configure the heap or not to collect garbage.

Program to Display Heap Memory Statistic

Memory is the most important resource in any application. System memories is one of tha main resource that an application developer has to consider while managing the application.

This is highly useful for the Application developer to get the information about the Total Heap Size, total memory been used by the application and the total free heap space.

Java's Runtime class gives lots of information about the resource details of the Java Virtual Machine(JVM). The memory consumed by the JVM can be read by different methods in Runtime class.

Below is show a simple program for getting the Total Heap Size, total Heap Memory Used and total free Heap Memory area.

Computer Code

/**
* Class: TestMemory
* @author: Rahul Sah
* @description: Prints JVM memory utilization statistics
*/
 public class TestMemory {
public static void main(String [] args) {

int mb = 1024*1024;

//Getting the runtime reference from system
Runtime runtime = Runtime.getRuntime();

System.out.println("##### Heap utilization statistics [MB] #####");

//Print used memory
 System.out.println("Used Memory:" + (runtime.totalMemory() - runtime.freeMemory()) / mb);

 //Print free memory
System.out.println("Free Memory:"+runtime.freeMemory() / mb);

 //Print total available memory
System.out.println("Total Memory:" + runtime.totalMemory() / mb);

 //Print Maximum available memory
System.out.println("Max Memory:" + runtime.maxMemory() / mb);

 }
}

How to Set Maximum Heap Size?

JVM used heap size to manage the Java application and for storing the object dynamically. If the Java Program requires a large amount of memory, it is possible that the virtual machine will throw OutOfMemoryError

exception.

Setting up the JVM heap Size

The JVM Heap Size can be increased at the runtime using command line options.

Computer Code

-Xms set initial Java heap size

-Xmx set maximum Java heap size

-Xss set java thread stack size

For Example setting up minimum heap to 64MB and maximum heap to 256MB for a Java Program "HelloWorld" can be done as
java -Xms64m -Xmx256m HelloWorld

Execution Engine

The bytecode that is assigned to the runtime data areas in the JVM via class loader is executed by the execution engine. The execution engine reads the Java Bytecode in the unit of instruction. It is like a CPU executing the machine command one by one. Each command of the bytecode consists of a 1-byte OpCode and additional Operand. The execution engine gets one OpCode and execute task with the Operand, and then executes the next OpCode.

But the Java Bytecode is written in a language that a human can understand, rather than in the language that the machine directly executes. Therefore, the execution engine must change the bytecode to the language that can be executed by the machine in the JVM. The bytecode can be changed to the suitable language in one of two ways.

1). Interpreter - Reads, interprets and executes the bytecode instructions one by one. As it interprets and executes instructions one by one, it can quickly interpret one bytecode, but slowly executes the interpreted result. This is the disadvantage of the interpret language. The 'language' called Bytecode basically runs like an interpreter.

2). JIT(Just-In-Time) - The JIT compiler has been introduced to compensate for the disadvantages of the interpreter. The execution engine runs as an interpreter first, and at the appropriate time, the JIT compiler compiles the entire bytecode to change it to native code. After that, the execution engine no longer interprets the method, but directly executes using native code. Execution in native code is much faster than interpreting instructions one by one. The compiled code can be executed quickly since the native code is stored in the cache.

However, it takes more time for JIT compiler to compile the code than for the interpreter to interpret the code one by one. Therefore, if the code is to be executed just once, it is better to interpret it instead of compiling. Therefore, the JVMs that use the JIT compiler internally check how frequently the method is executed and compile the method only when the frequency is higher than a certain level.

Class File Structure

The class file generated by Java Compiler is not human readable form but if open the class file and interpret it. It will follow a particular pattern. It has the following components in it:

The Diagrammatic representation of a Java Class file is been shown.

A Java Class file consist of 10 different components

1). Magic Number 0xCAFEBABE

2). Version of the Class File Format: the minor and major version of the Class File

3). Constant Pool : Pool of Constant for the Class

4). Access Flag : For example the access modifier of the class like static, public,private, abstract etc

5). This Class : The name of the current class

6). Super Class : The name of the super class

7). Interface Class : Any interface in the class

8). Fields : Any fields in the class

9). Methods : Any methods in the class

10). Attributes: Any attributes of the class (for example the name of the sourcefile, etc.)

You can remember the 10 sections with some funny mnemonic: My Very Cute Animal Turns Savage In Full Moon Areas.

Above diagram depicts that Java Class file is divided into different components such as Magic, Version, Constant Pool, Access Flaf, This Class, Super Class, Interfaces, Fields, Methods and Attributes.

The length of the Java class is not known before it gets loaded. There are variable length sections such as constant pool, methods, attributes etc. These sections are organized in such a way that they are prefaced by their size or length. This way JVM knows the size of variable length sections before actually loading them.

The data written in a Class file is kept at one byte aligned and is tightly packed. This helps in making class file compact.

The order of different sections in a Java Class file is strictly defined so that the JVM knows what to expect in a Class file and in which order it is loading different components.
